Key Question: Component Reuse Strategy

The packaging section already has advanced components (ItemNameCombobox, CategorySelector, ItemStatusDropdown). How should we approach this?

A

Adapt Existing Components In-Place

Modify the existing packaging components (ItemNameCombobox, CategorySelector, ItemStatusDropdown) to work in both the packaging form context AND the order item row context. Add props to control compact/inline mode vs modal mode.

Pros

  • True single source of truth
  • Any fix/improvement applies everywhere

Cons

  • More complex component APIs
  • Risk of breaking existing packaging form
B

Extract Shared Primitives + Compose

Extract the core search/select/create logic into shared primitives (e.g., SearchableSelect, ColorStatusPill), then have both PackagingForm and OrderItemRow compose from these primitives. Each context builds its own wrapper.

Pros

  • Clean separation of concerns
  • Each context controls its own layout
  • Easier to test primitives in isolation

Cons

  • More files to create
  • Some duplication in wrapper logic
C

Direct Reuse — Drop Existing Components Into Orders

Use the existing ItemNameCombobox, CategorySelector, and ItemStatusDropdown directly in OrderItemRow with minimal modifications. Add a "compact" or "inline" variant prop where needed for row layout.

Pros

  • Fastest to implement
  • Guaranteed identical behavior
  • Minimal new code

Cons

  • Components may not fit row layout perfectly
  • May need style overrides
