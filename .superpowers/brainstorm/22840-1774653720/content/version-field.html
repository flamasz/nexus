Version Field: How Should It Work?

Versions are currently plain strings (e.g., "v1", "v2") tied to an item_name + category combo. No dedicated Version table exists — they're derived from the items table.

How should "Create New Version" work?

When a user types a version that doesn't exist, what happens?

A

Free-text with suggestions

The version field becomes a searchable combobox. It shows existing versions for that item+category combo as suggestions, but the user can type any string (e.g., "v3", "R2", "Final"). The typed value is saved directly to the order item's version field.

No new DB table needed. Versions are just strings. "Create new" simply means typing a value that doesn't match existing ones.

B

Structured creation (like Item Name)

Add a "+ Create new version" option at the bottom of the dropdown. Clicking it shows a small inline form or confirmation. The new version string is validated and explicitly created. This mirrors the ItemNameCombobox pattern exactly.

No new DB table needed either — the "creation" is just accepting the new string value. But the UX has an explicit creation step rather than free-typing.
